# Локальный CRUD-сервис новостей (Sprint 1)

Проект представляет собой локальный бэкенд-сервис для управления новостями, разработанный в рамках стажировки Bitlab Academy. Фокус задачи — построение чистой слоистой архитектуры и реализация защищенного кода в соответствии со стандартами OWASP с самого первого спринта.

---

## 🛠️ Технологический стек
* **Java 21**
* **Spring Boot 3.x** (Spring Data JPA, Bean Validation)
* **PostgreSQL**
* **Dotenv Java** (для безопасного управления переменными окружения)

---

## 🚀 Инструкция по запуску локально

### 1. Подготовка базы данных
Убедитесь, что у вас установлен и запущен PostgreSQL (pgAdmin). Создайте пустую базу данных для проекта через SQL-консоль:
``sql
CREATE DATABASE news_db;

2. Настройка переменных окружения
В корневом каталоге проекта (в той же папке, где находится build.gradle) создайте файл под названием .env. Добавьте в него ваши локальные параметры подключения.

(Файл .env обязательно должен быть добавлен в .gitignore, чтобы ваши пароли не попали в публичный репозиторий):

Фрагмент кода
DB_URL=jdbc:postgresql://localhost:5432/news_db
DB_USERNAME=postgres
DB_PASSWORD=ваш_реальный_пароль_от_postgres

Реализованные меры безопасности (Secure Design)
В ходе разработки были учтены ключевые уязвимости из категории OWASP Top 10:

1. Защита от SQL Injection (ВЫСОКИЙ РИСК)
Решение: Параметризация запросов через Spring Data JPA.

Как это работает: Все операции взаимодействия с базой данных реализованы через абстракцию NewsRepository, наследующую JpaRepository. В NewsService используются встроенные методы (например, findById, save, deleteById). Прямая конкатенация строк для сборки SQL-запросов полностью исключена, что нейтрализует угрозу внедрения вредоносного SQL-кода.

2. Защита от Mass Assignment & Excessive Data Exposure (ВЫСОКИЙ/СРЕДНИЙ РИСК)
Решение: Архитектурное разделение внутренних сущностей базы данных (Entity) и объектов передачи данных (DTO).

Как это работает: Контроллер NewsController изолирован от слоя данных. На вход эндпоинты принимают строго NewsRequest, содержащий только разрешенные для изменения бизнес-поля (title, content, publishDate, author). На выход отдается объект NewsResponse. Прямой биндинг сущностей из тела запроса исключен, что не позволяет злоумышленнику подменить id или перезаписать системные свойства.

3. Валидация входных данных (НИЗКИЙ РИСК)
Решение: Проверка входящих параметров на корректность на уровне DTO при помощи Bean Validation (jakarta.validation).

Как это работает: Поля класса NewsRequest защищены аннотациями:

@NotBlank и @Size(min = 3, max = 100) гарантируют, что заголовок не будет пустым или слишком коротким/длинным.

@PastOrPresent предотвращает отправку новостей с датой публикации в будущем.

В контроллере перед @RequestBody установлена аннотация @Valid, которая автоматически прерывает запрос и возвращает ошибку 400 Bad Request при нарушении правил.

4. Защита от утечки данных в ошибках (СРЕДНИЙ РИСК)
Решение: Использование глобального перехватчика исключений @ControllerAdvice и маскирование системных трейсов.

Как это работает: Класс GlobalExceptionHandler перехватывает общие ошибки Exception.class. Внешний клиент получает стерильный JSON-ответ со статусом 500 и общим сообщением "Произошла внутренняя ошибка сервера". При этом реальный технический стектрейс (включая детали SQL/JPA) логируется с помощью slf4j исключительно во внутреннюю консоль сервера для разработчика. Дополнительно в application.yml выставлено правило include-stacktrace: never.

5. Безопасное хранение секретов
Решение: Вынос учетных данных БД из конфигурационных файлов в переменные окружения.

Как это работает: Файл конфигурации application.yml не содержит захардкоженных паролей. Вместо них используются подстановки ${DB_URL}, ${DB_USERNAME}, ${DB_PASSWORD}. Инициализация переменных происходит на старте в классе NewsApplication динамически через библиотеку cdimascio.dotenv.

6. Защита заголовков сервера (Разведка / Reconnaissance)
Решение: Скрытие и маскировка сигнатуры встроенного контейнера Tomcat.

Как это работает: В файле application.yml переопределен параметр server.server-header: "Server". Это полностью удаляет из HTTP-заголовков ответов дефолтную строку Apache-Coyote/1.1, скрывая точную версию используемого ПО и усложняя злоумышленнику поиск известных уязвимостей (CVE) под конкретную версию веб-сервера.

Тестирование API
Для быстрой и удобной проверки работоспособности всех эндпоинтов в корне проекта подготовлен файл requests.http. Он содержит готовые сценарии для проверки всех 4 CRUD-операций, а также готовые негативные тесты для проверки безопасности (тест валидации на ошибку 400 и тест отсутствия записи на ошибку 404). Запустить их можно напрямую через встроенный HTTP-клиент в IntelliJ IDEA.
